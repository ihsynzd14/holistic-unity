## F37 — `charge.refunded` → requires_manual_refund=false

### Background: come funziona il flag `requires_manual_refund`

Setto da [`therapist-webapp/api/bookings/[id]/cancel/route.ts:273-280`](../08_Codebases/therapist-webapp/src/app/api/bookings/[id]/cancel/route.ts) (logic 240-254):

```ts
// Flag manual review for post-payout, non-success status, or short refund.
const expectedCents = Math.round(Number(booking.price) * 100);
if (isPostPayout) {                                  // chargeAgeDays > 14
  requiresManualRefund = true;
  manualRefundNote = `Refund issued ${chargeAgeDays}d after charge…`;
} else if (refund.status !== "succeeded") {          // Stripe ha 'pending' o 'failed'
  requiresManualRefund = true;
  manualRefundNote = `Stripe refund status is "${refund.status}" — verify completion.`;
} else if (refund.amount < expectedCents) {          // partial refund
  requiresManualRefund = true;
  manualRefundNote = `Refunded ${refund.amount}¢ but client paid ${expectedCents}¢ — short.`;
}
```

**Schema**: columns `bookings.requires_manual_refund (boolean DEFAULT false)` + `bookings.manual_refund_note (text)` + view `admin_pending_manual_refunds` aggiunti via Dashboard SQL (memory.md riga 256). **Migration NON versionata in git** (è un debt separato — vedi gap nei task list "untracked migrations").

### Code trace PRE-FIX ([stripe-webhook/index.ts:582-625](../08_Codebases/iOS_App/supabase/functions/stripe-webhook/index.ts))

| Step | Riga | Cosa fa |
|------|------|---------|
| 1 | 584-585 | Estrae `charge.payment_intent` |
| 2 | 588-590 | Calcola `refundedAmount`, `totalAmount`, `isFullRefund` |
| 3 | 592-602 | UPDATE transactions SET status='refunded'/'partially_refunded', refund_amount |
| 4 | 605-617 | **Se full refund**: UPDATE bookings SET status='cancelled' — **NON toccava `requires_manual_refund`** ❌ |

### Bug identificato

Scenario:
1. Terapista cancella un booking 10gg prima della seduta (within 14d window ✓)
2. `/api/bookings/[id]/cancel` chiama Stripe `refunds.create` con `reverse_transfer + refund_application_fee`
3. Stripe ritorna refund con `status="pending"` (perché alcuni metodi di pagamento processano async, es. SEPA/bancomat)
4. Cancel route flagga `requires_manual_refund=true, manual_refund_note="Stripe refund status is 'pending' — verify completion"`
5. Booking entra in vista `admin_pending_manual_refunds`
6. Stripe processa il refund con successo entro 1-5 giorni
7. Stripe fires `charge.refunded` webhook
8. **Webhook updatava transaction MA NON il booking flag** → booking rimane in `admin_pending_manual_refunds` FOREVER, anche se Stripe ha confermato il successo

### Fix applicato

[`stripe-webhook/index.ts:605-617`](../08_Codebases/iOS_App/supabase/functions/stripe-webhook/index.ts) — aggiunte 3 righe + commento esplicativo:

```diff
   if (isFullRefund) {
     const { data: tx } = await supabaseAdmin
       .from("transactions")
       .select("booking_id")
       .eq("stripe_payment_intent_id", paymentIntentId)
       .maybeSingle();

     if (tx?.booking_id) {
       await supabaseAdmin
         .from("bookings")
-        .update({ status: "cancelled" })
+        .update({
+          status: "cancelled",
+          requires_manual_refund: false,
+        })
         .eq("id", tx.booking_id);
     }
   }
```

### Trade-off accettato

Il fix clear `requires_manual_refund` su QUALSIASI full refund, anche se il flag era stato originariamente settato per:
- **(a)** post-payout window (>14gg)
- **(c)** short refund amount

Per questi 2 casi, l'admin dovrebbe ANCHE verificare:
- Caso (a): bilancio del connected account su Stripe Dashboard (potrebbe essere andato negativo)
- Caso (c): l'importo refunded vs expected su Stripe Dashboard

Queste verifiche **non sono perfettamente specchiate nel nostro DB** — il flag `requires_manual_refund` non distingue tra i 3 motivi. Per V2 si potrebbe aggiungere un enum `manual_refund_reason` (`'pending_verify'|'post_payout'|'short_amount'`) per discriminare quali motivi devono persistere oltre il webhook charge.refunded.

**Per ora** (MVP): accettiamo che il flag venga cleared dal webhook charge.refunded. L'admin sa che deve periodicamente fare quick-scan delle cancellazioni post-payout via Stripe Dashboard, indipendentemente dal flag (documentato in `manual_refund_note` mantenuto + admin training).

### Verdict F37

⚠️ **FAIL → FIXED**. Pre-deploy: `supabase functions deploy stripe-webhook` da workdir `08_Codebases/iOS_App/`.
