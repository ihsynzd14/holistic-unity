TASK: CONTROL CHECK — verifica esaustività Gate 1/2/3 in AuthManager.resolveAuthState()

PROBLEMA:
Il CONTROL CHECK richiedeva la verifica che i gate nel metodo
`resolveAuthState(for:)` di AuthManager.swift fossero logicamente esaustivi,
ossia che ogni combinazione possibile di input (authProvider, isEmailVerified,
role, clientOnboarded, needsTOS) producesse uno stato definito senza
fall-through.

ANALISI:
Enumerati tutti i possibili percorsi di input tramite truth table.
I 4 gate (email verification → role assignment → onboarding → TOS acceptance)
coprono tutti i casi: nessun input produce uno stato indefinito.
- Gate 1 (email verification): corretto — Apple/Google bypassano, email users
  bloccati se `isEmailVerified == false`.
- Gate 2 (role): cattura tutti gli utenti senza ruolo, qualsiasi provider.
- Gate 3 (onboarding): verificato per client; therapist esenti per design
  (onboardano via web portal, redirect-out gestito da AppCoordinator).
- Gate 4 (TOS): gestisce entrambi i ruoli.
- Race condition protette con `if case .authenticated = self.authState`
  prima di ogni downgrade async.
- `hasCompletedClientOnboarding` fails-open su errore rete (trade-off
  documentato per evitare lockout durante outage).

Esito: PASS — gate logicamente esaustivi.

FIX ESEGUITO:
- AuthManager.swift: aggiornato doc-comment di `resolveAuthState(for:)`
  (gate 3): il commento diceva "handled elsewhere" ma il client onboarding è
  ora gestito direttamente qui (aggiunto 2026-05-18). Riscritto per chiarire
  che Gate 3 vale solo per client (`.needsOnboarding(.client)`), usa
  `hasCompletedClientOnboarding` con fail-open, e i therapist sono esenti
  con redirect-out in `AppCoordinator`.
