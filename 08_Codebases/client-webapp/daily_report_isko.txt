TASK: CONTROL CHECK — verifica esaustività Gate 1/2/3 in AuthManager.resolveAuthState()

PROBLEMA:
Il CONTROL CHECK richiedeva la verifica che i gate nel metodo
`resolveAuthState(for:)` di AuthManager.swift fossero logicamente esaustivi,
ossia che ogni combinazione possibile di input (authProvider, isEmailVerified,
role, clientOnboarded, needsTOS) producesse uno stato definito senza
fall-through.

ANALISI:
Enumerati tutti i possibili percorsi di input tramite truth table.
I 4 gate (email verification → role assignment → onboarding → TOS acceptance)
coprono tutti i casi: nessun input produce uno stato indefinito.
- Gate 1 (email verification): corretto — Apple/Google bypassano, email users
  bloccati se `isEmailVerified == false`.
- Gate 2 (role): cattura tutti gli utenti senza ruolo, qualsiasi provider.
- Gate 3 (onboarding): verificato per client; therapist esenti per design
  (onboardano via web portal, redirect-out gestito da AppCoordinator).
- Gate 4 (TOS): gestisce entrambi i ruoli.
- Race condition protette con `if case .authenticated = self.authState`
  prima di ogni downgrade async.
- `hasCompletedClientOnboarding` fails-open su errore rete (trade-off
  documentato per evitare lockout durante outage).

Esito: PASS — gate logicamente esaustivi.

FIX ESEGUITO:
- AuthManager.swift: aggiornato doc-comment di `resolveAuthState(for:)`
  (gate 3): il commento diceva "handled elsewhere" ma il client onboarding è
  ora gestito direttamente qui (aggiunto 2026-05-18). Riscritto per chiarire
  che Gate 3 vale solo per client (`.needsOnboarding(.client)`), usa
  `hasCompletedClientOnboarding` con fail-open, e i therapist sono esenti
  con redirect-out in `AppCoordinator`.

TASK: CONTROL CHECK — VideoCallService reconnection logic (rete che cade)

PROBLEMA:
Il CONTROL CHECK richiedeva verifica che la logica di riconnessione di
LiveKit gestisse correttamente la caduta della rete durante una sessione
video attiva.

ANALISI:
- Caduta breve (LiveKit SDK internal retry): delegate forwarding a
  callState=.reconnecting, poi .connected. ✅
- Caduta lunga: transizione a .failed via transitionToFailed con
  cancellazione di timer/heartbeat/qualityMonitor. ✅
- UserDefaults SR-01 persistence per app-crash recovery. ✅
- Race guard pre-reconnect `guard callState == .reconnecting else { return }`. ✅
- Token refresh automatico su 401/JWT errors. ✅
- AVAudioSession reset on disconnect. ✅

BUG REALE rilevato:
`reconnectAttempts` veniva resettato a 0 dentro `connect()` (l.139), e
poiché `reconnectWithBackoff()` chiama `disconnect()` + `connect()`,
il counter veniva azzerato ad ogni tentativo dell'utente. Conseguenze:
- Exponential backoff non funzionava (sempre 2s, mai 4s/8s)
- Cap "max 3 attempts" mai raggiunto (utente poteva cliccare infinite volte)
- UI button label "Try Again (N left)" mostrava flapping 3→2→3→2
- `actualConnectedSeconds` aveva lo stesso bug: sessioni 25min che si
  riconnettevano al min 24 venivano falsamente flaggate come failed
  (BL-02) perché il counter veniva resettato.

INTERVENTI ESEGUITI:
- VideoCallService.swift: aggiunto parametro `isReconnect: Bool = false`
  a `connect(...)`. Spostati i reset di `elapsedSeconds`,
  `actualConnectedSeconds` e `reconnectAttempts` dentro `if !isReconnect`.
  `ghostConnectionDetected` e `hasSeenRemoteParticipant` restano fuori
  (devono resettarsi anche su reconnect per evitare falsi positivi di
  "partecipante uscito").
- `reconnectWithBackoff()` ora chiama `connect(..., isReconnect: true)`.
- Doc-comment aggiunto al parametro che spiega il motivo del fix.
- Backward compat preservata: l'unico external caller (VideoCallView.swift:146)
  usa il default `isReconnect: false`, comportamento per nuove sessioni
  invariato.

Esito: PARTIAL PASS → PASS dopo fix.

PROBLEMA:
Il CONTROL CHECK richiedeva che gli errori di pagamento (Stripe) e di
prenotazione fossero mostrati all'utente con messaggi user-friendly in
italiano. L'audit ha rilevato 4 problemi sovrapposti:

2. Gli errori Stripe server-side venivano passati through letterali

   ecc.) — nessun mapping a italiano.
3. Le 4 entry "Booking failed: %@", "Could not prepare payment: %@",
   "Payment failed: %@", "Payment cancelled..." erano in Localizable.xcstrings
   ma SENZA traduzione italiana — solo l'inglese veniva mostrato.

   non passate per `String(localized:)`.

  `decline_code` JSON, fallback su substring matching del testo

- SupabasePaymentRepository.swift: 6 call site di `PaymentError.paymentFailed(...)`
  ora usano o `String(localized:)` o `StripeErrorMapper.*`. Rimossi tutti
  i string literal inglesi. Riordinato il blocco di gestione errore HTTP

- SupabasePaymentRepository.swift / PaymentError: 3 case localizzate via

- SupabaseBookingRepository.swift / BookingError: 3 case localizzate via
  `String(localized:)`.
- ManageBookingView.swift: 6 stringhe hardcoded → tutte avvolte in
  `String(localized:)` con commenti contestuali.
- Localizable.xcstrings: aggiunte traduzioni italiane alle 4 entry
  esistenti senza `localizations`. Aggiunte 23 NUOVE entry complete
