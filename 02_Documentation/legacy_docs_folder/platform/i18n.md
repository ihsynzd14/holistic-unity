# Internationalization (i18n)

**Last verified:** 2026-04-16 by Marcello
**Status:** ✅ EN + IT complete for V1 iOS · ⚠️ pt-BR partial · ✅ Webapp has own i18n system
**Owner:** Marcello

## Purpose

Holistic Unity ships in **Italian and English** for V1. The platform auto-selects the user's language from device locale — no in-app toggle. The goal is that an Italian user sees fully Italian UI including greetings, section titles, date connectors, error messages, and rotating daily intentions; an English user sees the same native-feel in English.

Content authored by therapists (service names, bio, reviews) is **not** translated — it stays in the language the therapist wrote it. This is intentional: a client booking a therapist whose bio is in Italian should be able to read the actual words the therapist chose.

## Surfaces

| Surface | Source language | Translation mechanism | Files |
|---------|------------------|------------------------|-------|
| **iOS app** | English (source) | `Localizable.xcstrings` + `String(localized:)` | `Holistic Unity/Localizable.xcstrings` |
| **therapist-webapp** | English (source) | Custom React i18n context | `src/lib/i18n/translations/{en,it}.ts` |
| **admin-dashboard** | English (source) | (Same as webapp — in practice Italian-only today) | Same pattern |

Device locale detection is automatic on iOS (`Locale.current` / `Bundle.main.preferredLocalizations`). On the webapp it's set via `useI18n()` context with a locale state persisted in localStorage.

## iOS — how strings work

### The `.xcstrings` catalog

Xcode 15+ uses a single JSON catalog: `Localizable.xcstrings`. Every `Text("...")` or `String(localized: "...")` call has its source string extracted at build time as a catalog key. Translations live alongside each key in a nested structure:

```json
{
  "sourceLanguage": "en",
  "strings": {
    "Upcoming Sessions": {
      "localizations": {
        "it": { "stringUnit": { "state": "translated", "value": "Le tue prossime sessioni" }}
      }
    }
  }
}
```

`state` values:
- `new` — key extracted but never translated (iOS shows source language fallback)
- `translated` — curated by a human, ready for release
- `needs_review` — translator flagged it
- `stale` — source changed after translation, retranslate

Today's stats (from `Localizable.xcstrings`):
- 430+ keys total
- **EN: sourceLanguage (always filled)**
- **IT: ~390 translated**
- pt-BR: ~32 partial (safe to ignore V1)

### Pattern for developers

Always use `String(localized:)` when a user-facing string is not already a plain SwiftUI `Text("...")`. SwiftUI's `Text` auto-localizes from the catalog at render time, but pure `String` variables don't — they need the explicit init.

✅ **Do:**

```swift
// SwiftUI Text (auto-localized)
Text("Welcome back")

// Pure Swift string
let title = String(localized: "Choose The Best Therapy",
                   comment: "Categories section title default")
```

❌ **Don't:**

```swift
// Hardcoded literal — will NOT be translated
let title = "Choose The Best Therapy"

// String interpolation can't be extracted as a stable key
let greeting = "\(timeGreeting), \(firstName)"  // OK only if parts are each localized
```

### Dynamic content

When you need interpolation with multiple localized pieces, localize each piece separately then compose. See `PersonalizationEngine.greeting(...)` for the canonical example:

```swift
let timeGreeting = switch hour {
    case 5..<12:  String(localized: "Good morning", comment: "Home greeting, 5am–12pm")
    case 12..<17: String(localized: "Good afternoon", comment: "Home greeting, 12pm–5pm")
    ...
}
return "\(timeGreeting), \(firstName)"  // composition is locale-neutral
```

For date/time connectors that differ across locales (`at` → `alle`), localize the connector itself:

```swift
// Booking.formattedDateTime
let connector = String(localized: "at",
                       comment: "Connector between date and time, e.g. 'Apr 16 at 9:30'")
return "\(formattedDate) \(connector) \(formattedTime)"
```

iOS `Date.formatted(...)` already uses locale-specific month/day ordering automatically. Don't reinvent.

## Adding a new string

1. Write the English source in code:

   ```swift
   Text("Your new label")
   // or
   String(localized: "Your new label", comment: "Context for the translator")
   ```

2. Build the project. Xcode extracts the new key into `Localizable.xcstrings`.

3. Open the catalog in Xcode (double-click the `.xcstrings` file). Add the Italian translation inline.

   Or do it in bulk via JSON (see `/tmp/translate_xcstrings.py` pattern — a simple dict keyed by EN source).

4. Rebuild. Italian locale now shows translated text.

## Adding a new language

1. Open `Localizable.xcstrings` in Xcode → click `+` under "Localizations" in the catalog editor → pick the language.
2. Every key now has an empty slot for that language.
3. Fill them in (Xcode editor, CSV export, or bulk-update via JSON).
4. Update `Info.plist` if you want to offer in-app language switching (V1.1 — today it's device-locale only).

## Translation quality & review

Translations for V1 were written by hand (not Google Translate). They try to be:

- **Natural** — not literal (`"Trust the rhythm of your own season"` → `"Fidati del ritmo della tua stagione"`, not `"Fiducia il ritmo della propria stagione"`).
- **Tone-consistent** — wellness-appropriate, warm but not saccharine. No imperative verbs unless the source uses them.
- **Gender-neutral where possible** — `"Sei sicuro"` is a known shortcoming; in a V1.1 pass we could adopt `"Sei certa/o"` or rewrite to avoid gendered forms.

Before public launch it's highly recommended to have a native Italian copywriter review the ~390 IT strings. Reasonable cost ≈ 2–3 hours. Export via:

```bash
python3 -c "
import json
with open('Holistic Unity/Localizable.xcstrings') as f: d = json.load(f)
import csv, sys
w = csv.writer(sys.stdout)
w.writerow(['key', 'en', 'it', 'state'])
for k, v in d['strings'].items():
    it = v.get('localizations', {}).get('it', {}).get('stringUnit', {})
    w.writerow([k, k, it.get('value', ''), it.get('state', 'new')])
" > /tmp/hu-translations-review.csv
```

## Known caveats

### 1. Date & number formatting
iOS's `Date.formatted(date: .abbreviated, ...)` already produces locale-specific output (`Apr 16` in EN, `16 apr` in IT). Don't build custom date formatters unless necessary. Currency follows `Locale.current.currency`.

### 2. Plurals
`.xcstrings` supports plural variants via `variations.plural.one`, `.other`. We don't use them yet (V1 has some naive "1 session left" / "%lld sessions left" cases). V1.1 cleanup: convert these to proper plural forms.

### 3. Push notification payload
Server-sent notifications (from `send-push-notification` edge function) are currently **EN only**. To localize them:
- Option A: the server sends a `loc_key` (e.g. `"new_booking_notif"`) → client Info.plist has matching localized `loc_key` entries.
- Option B: server reads `users.preferred_language` and picks IT/EN at send time.

We'll do Option B in V1.1 (simpler, server controls message body).

### 4. Content from the DB is passthrough
Service names (`"Sessione di 60 Minuti"`), therapist bios, review text — all user-generated, never translated. A user in Italy booking an English therapist sees the English service name. This is correct.

### 5. Info.plist permission strings
`NSCameraUsageDescription`, `NSMicrophoneUsageDescription`, etc. are currently English-only in `Holistic-Unity-Info.plist`. To localize, create per-language `InfoPlist.strings` files (`en.lproj/InfoPlist.strings`, `it.lproj/InfoPlist.strings`). V1.1 cleanup.

## Webapp — separate system

`therapist-webapp/src/lib/i18n/translations/{en,it}.ts` is a plain TypeScript map used by a React context (`useI18n()`). Pattern:

```typescript
// en.ts
export default {
  nav: { home: "Home", ... },
  login: { title: "Holistic Unity", ... },
};

// in component
const { t } = useI18n();
return <h1>{t.login.title}</h1>;
```

Locale is read from `localStorage.getItem("locale")` on mount (default `it` because most therapist users are Italian). There's no auto-detect from browser locale — the therapist picks once.

Today `it.ts` is canonical (Italian-first dashboard), `en.ts` is a translated counterpart. About 80% coverage across both.
